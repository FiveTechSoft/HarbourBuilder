=== BENCHMARK COMPARISON: IDE C++ Core vs FiveWin ===
Date: 2026-03-31
Machine: Intel Xeon W-2140B @ 3.20GHz, 32GB RAM, Windows 11

============================================================
TEST 1: Create 500 buttons
============================================================
  FiveWin:    0.2430 sec  (   2,057 controls/sec)
  IDE C++:    0.0010 sec  ( 499,999 controls/sec)
  >> IDE is ~243x FASTER

============================================================
TEST 2: Set property 100,000 times
============================================================
  FiveWin:   24.8610 sec  (   4,022 sets/sec)  [nLeft := n, triggers SetWindowPos]
  IDE C++:    0.0700 sec  (1,428,571 sets/sec)  [UI_SetProp + SetWindowPos]
  >> IDE is ~355x FASTER

============================================================
TEST 3: Get property 100,000 times
============================================================
  FiveWin:    0.0470 sec  (2,127,659 gets/sec)  [Harbour DATA access]
  IDE C++:    0.0600 sec  (1,666,666 gets/sec)  [HB_FUNC + C++ field]
  >> FiveWin is ~1.3x faster (Harbour DATA is direct memory, no function call)

============================================================
TEST 4: Get 2 properties 100,000 times (OOP wrapper)
============================================================
  FiveWin:    0.0830 sec  (2,409,638 gets/sec)  [oBtn:nLeft + oBtn:nTop]
  IDE OOP:    0.1690 sec  (1,183,431 gets/sec)  [oBtn:Left + oBtn:Top via ACCESS]
  >> FiveWin is ~2x faster (direct DATA vs ACCESS+HB_FUNC bridge)

============================================================
ANALYSIS
============================================================

CREATION (Test 1): IDE is 243x faster.
  FiveWin's TButton:New() has 20+ parameters, extensive DEFAULT processing,
  coordinate conversion, font handling, AddControl, etc.
  IDE's UI_ButtonNew() does: new TButton() + set 4 fields + AddChild. Minimal.

SET PROPERTY (Test 2): IDE is 355x faster.
  FiveWin's nLeft:= triggers the ASSIGN method which calls SetWindowPos
  every time, even before the window exists. The overhead is in Harbour
  method dispatch + redundant Win32 calls.
  IDE's UI_SetProp sets the C++ field directly. SetWindowPos only if FHandle exists.
  NOTE: FiveWin's 24.86 sec suggests it's calling SetWindowPos 100K times
  on a live window, which involves actual window repositioning each time.

GET PROPERTY (Tests 3-4): FiveWin is 1.3-2x faster.
  This is expected: FiveWin's DATA is a direct Harbour instance variable
  (single memory read). IDE's bridge goes through HB_FUNC → C++ field → return,
  which has function call overhead. For pure property reads, Harbour OOP wins.

CONCLUSION:
  - For CREATION and MODIFICATION: IDE C++ core is 100-350x faster
  - For READ-ONLY access: FiveWin is slightly faster (direct DATA)
  - Overall for GUI work (create + move + resize + repaint): IDE wins massively
  - The bottleneck in real apps is creation + SetWindowPos, where IDE dominates
